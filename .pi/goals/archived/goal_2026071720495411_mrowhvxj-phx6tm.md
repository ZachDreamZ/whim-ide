# Goal Prompt

i need you to fix the workflow of the agents make sure they are not like overstepping each other also Remove the hard-coded tool-iteration limit that causes this error:

“Agent error
Stopped after 30 tool iterations (possible runaway loop). Review the plan and run again.”

CURRENT PROBLEM
The agent is forcibly stopped after 30 tool calls even when it is still making valid progress and the main/parent agent is continuing to provide instructions. This prematurely terminates long tasks such as codebase analysis, implementation, testing, debugging, visual validation, and repair loops.

REQUIRED BEHAVIOR

1. Remove the fixed 30-iteration termination
- Do not stop an active agent merely because it reached 30 tool iterations.
- Search the entire codebase for:
  - `30`
  - `maxIterations`
  - `maxToolIterations`
  - `maxSteps`
  - `iterationLimit`
  - `runaway`
  - `toolCallCount`
  - `stepCount`
  - the exact displayed error message
- Identify every frontend, backend, runtime, orchestration, and persistence path enforcing this limit.

2. Parent/main agent controls completion
A child or worker agent should continue operating while:
- The parent/main agent continues assigning work.
- The task has unresolved instructions.
- Valid tool calls are still being produced.
- The user has not cancelled the task.
- No fatal or unrecoverable error has occurred.

The worker must stop when:
- The main agent explicitly returns a final completion state.
- The parent agent sends a stop/cancel/abort instruction.
- The user cancels the task.
- The process encounters an unrecoverable error.
- The application shuts down or the task is externally terminated.

3. Do not interpret temporary pauses as completion
The worker must not stop simply because:
- One model response contains no tool call.
- A subtask finishes while the parent still has remaining work.
- A test fails and another repair attempt is needed.
- A tool returns an empty result.
- The agent has performed many valid actions.
- The agent needs to inspect, edit, test, and re-test repeatedly.

4. Replace the fixed cap with lifecycle states
Use explicit execution states such as:

- running
- waiting_for_parent
- waiting_for_tool
- retrying
- completed
- cancelled
- failed

The run loop should conceptually behave like:

while not cancelled and not completed and not fatally_failed:
    receive next parent/model instruction
    execute requested tool actions
    return results to parent
    continue until parent explicitly completes the task

The iteration count may still be recorded for telemetry, but it must not automatically terminate the task.

5. Preserve user cancellation
Removing the iteration cap must not make the application impossible to stop.

Keep or add:
- A functional Stop button.
- AbortController or equivalent cancellation propagation.
- Cancellation of active tool calls where supported.
- Cancellation propagation from parent agent to child agents.
- Cleanup of subprocesses, browser sessions, observers, and pending promises.
- Clear task status after cancellation.

6. Prevent genuine infinite loops without a small arbitrary cap
Do not replace 30 with another small fixed number such as 50, 100, or 500.

Instead, detect genuine non-progress loops based on behavior:
- Identical tool call repeated with identical arguments and identical result.
- Repeated unchanged plan with no new state.
- No files, outputs, test results, task state, or observations changing across many cycles.
- Repeated retry of the same failure without modifying the attempted solution.
- Parent-child agents continuously returning the same instruction to each other.

When a possible loop is detected:
- Do not immediately kill the run.
- Send the loop evidence back to the main agent.
- Allow the main agent to revise the approach.
- Only terminate when the parent explicitly decides it cannot recover, the user cancels, or a fatal condition occurs.

7. Configuration
If an iteration safety setting must remain for compatibility:
- Make it configurable.
- Support `0`, `null`, or `unlimited` as no iteration limit.
- Set the default to unlimited for normal agent tasks.
- Do not silently fall back to 30.
- Clearly distinguish an optional administrator limit from the normal agent lifecycle.

Example configuration:

maxToolIterations: null
maxAgentSteps: null
stopPolicy: "parent-controlled"

8. Nested-agent behavior
For delegated tasks:
- The main agent owns the overall run.
- Child agents report results back to the parent.
- A child completing its assigned subtask must not terminate the parent run.
- A child reaching a historical iteration threshold must not terminate the parent run.
- Parent cancellation must propagate to all children.
- Child failures should be reported to the parent so it can retry, replace, or revise the subtask.

9. UI behavior
Remove the misleading 30-iteration error state.

During long runs, display useful status such as:
- Current task or subtask
- Tool currently running
- Number of completed actions
- Elapsed time
- Stop control
- Waiting/retrying state

Do not display a failure merely because the action count is high.

10. Persistence and resume
Ensure long-running task state is not corrupted when:
- The UI re-renders.
- The user changes panels.
- A tool call takes a long time.
- The application reconnects to the backend.
- The parent agent delegates multiple sequential subtasks.

The iteration counter must not be restored as a termination trigger after restart or reconnection.

MANDATORY TESTS

Test 1: More than 30 valid tool calls
- Run a controlled task that requires at least 40–60 tool calls.
- Confirm it continues past iteration 30.
- Confirm no old error appears.

Test 2: Parent-directed completion
- Have the parent assign several sequential subtasks.
- Confirm the worker continues until the parent explicitly declares completion.

Test 3: Child-agent completion
- Complete one delegated child task.
- Confirm the parent continues with later subtasks.

Test 4: Manual cancellation
- Start a long-running task.
- Press Stop.
- Confirm the model loop, active tools, child agents, and subprocesses terminate cleanly.

Test 5: Repeated-tool loop
- Simulate identical repeated tool calls.
- Confirm loop evidence is reported to the parent rather than stopping at an arbitrary iteration count.
- Confirm the parent can change strategy and continue.

Test 6: Failure and repair loop
- Introduce a test failure.
- Confirm the agent can inspect, modify, rerun, and continue beyond 30 total actions.

Test 7: Restart/reconnect
- Start a long task, reconnect the UI, and confirm the task continues without restoring the old 30-step cutoff.

IMPLEMENTATION CONSTRAINTS
- Fix the actual orchestration/runtime logic, not only the visible error message.
- Do not simply change `30` to a larger number.
- Do not remove cancellation or fatal-error handling.
- Do not allow detached child agents to continue after the parent or user cancels.
- Add regression tests covering the previous 30-iteration failure.
- Run type checks, automated tests, build validation, and an end-to-end long-task test.

FINAL REPORT
After implementation, report:
- The root cause.
- Every file changed.
- Every old iteration-limit path removed or updated.
- The new stop/completion lifecycle.
- How parent and child cancellation works.
- Test evidence proving the agent continued beyond 30 tool iterations.
- Confirmation that it stops only on parent completion, explicit cancellation, fatal failure, or shutdown.

Do not ask me for clarification. Inspect the implementation, remove the fixed iteration cutoff, implement parent-controlled execution, and validate it end to end.
